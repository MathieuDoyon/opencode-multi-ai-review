# Model Family Selection Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Keep `/multi-review` model selection concise by showing one recommended model per family, with an advanced option for variant overrides.

**Architecture:** This is a prompt-template change, not runtime TypeScript logic. `commands/multi-review.md` will instruct the opencode agent to group `opencode models` output, pick recommended family defaults, ask a short normal question, and ask follow-up variant questions only when advanced selection is requested. `src/package-docs.test.ts` will assert the command template documents the required selection behavior.

**Tech Stack:** Markdown opencode command template, TypeScript, Vitest, pnpm.

## Global Constraints

- Do not add runtime model-grouping code unless the command-template approach proves insufficient.
- Preserve provider namespace when grouping models, so `opencode/mimo` and `opencode-go/mimo` stay separate.
- Treat OpenAI `gpt-5.x` minor lines as distinct product families, so `openai/gpt-5.4` and `openai/gpt-5.5` both remain visible.
- Prefer recent stable/full variants over reduced/specialized variants such as `mini`, `fast`, `flash`, `free`, and `spark`, unless the family only has those variants.
- Never pass the advanced sentinel label to `multi_ai_code_review`; pass exact model IDs only.
- Do not commit unless the user explicitly asks for a commit in the active session.

- [ ] **Step 3: Replace the command template instructions**

Replace the full contents of `commands/multi-review.md` with:

```md
---
description: Run a multi-model AI code review for the current branch
---

Run a multi-model code review of the current branch against its base ref.

Arguments from the user, if present: `$ARGUMENTS`

Steps:
1. Read the model list below, generated by `opencode models`.
2. Group the available model IDs by model family before asking the user which models to use.
   - Preserve provider namespace, so `opencode/mimo` and `opencode-go/mimo` are separate families.
   - Treat model name roots as families, so `glm-5.1` and `glm-5.2` group as `opencode-go/glm`.
   - Treat OpenAI `gpt-5.x` minor lines as distinct product families, so `openai/gpt-5.4` and `openai/gpt-5.5` both remain visible.
   - Prefer the newest stable/full variant in each family over reduced or specialized variants such as `mini`, `fast`, `flash`, `free`, and `spark`, unless the family only has those variants.
3. Ask the user: `Which model families should review this branch?`
   - Explain: `I picked one recommended model per family to keep the list short. Select "Advanced: choose specific model variants" if you want to override variants.`
   - Use the built-in `question` tool with `multiple: true`.
   - Include one recommended model per family as choices.
   - Include `Advanced: choose specific model variants` as the final choice when any family has multiple variants.
4. If the user selects `Advanced: choose specific model variants`, ask concise follow-up questions only for families with multiple variants or families the user wants to override. Avoid showing every model in one large list unless grouping is not possible.
5. Build the final exact model ID list.
   - Never pass the advanced option label to `multi_ai_code_review`.
   - Pass only exact selected model IDs in `models`.
6. If `$ARGUMENTS` contains a token that looks like a git ref such as `origin/main`, pass it as `baseRef`. Pass the remaining argument text as `instructions`.
7. Call the `multi_ai_code_review` tool with the selected `models`, optional `baseRef`, and optional `instructions`.
8. Return the Markdown report from the tool without rewriting its table.

Available models:
!`opencode models`
```
